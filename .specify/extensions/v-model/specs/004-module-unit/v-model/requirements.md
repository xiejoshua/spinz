# V-Model Requirements Specification: Module Design ↔ Unit Testing

**Feature Branch**: `004-module-unit-testing`  
**Created**: 2026-02-21  
**Status**: Approved  
**Source**: `specs/004-module-unit/spec.md`

## Overview

This document formalizes the requirements for extending the V-Model Extension Pack from Architecture Design ↔ Integration Testing (v0.3.0) down to Module Design ↔ Unit Testing (v0.4.0) — the bottom of the V-Model, where specification meets code. The extension adds two new commands (`/speckit.v-model.module-design` and `/speckit.v-model.unit-test`), a deterministic validation script with `[EXTERNAL]` bypass and partial validation logic, extended traceability matrix support (Matrix D), and corresponding templates. Module design follows DO-178C Low-Level Requirements and ISO 26262 Part 6 with four mandatory views (Algorithmic/Logic with fenced pseudocode blocks, State Machine with Mermaid stateDiagram-v2, Internal Data Structures, Error Handling) and a Target Source File(s) mapping that bridges specification to codebase. Unit testing follows ISO/IEC/IEEE 29119-4 white-box techniques (Statement & Branch Coverage, Boundary Value Analysis with Equivalence Partitioning fallback for non-scalar types, Strict Isolation via Dependency & Mock Registry including hardware interfaces, State Transition Testing). Safety-critical additions (MISRA C complexity constraints, MC/DC structural coverage, variable-level fault injection) are conditional on domain configuration. The `[CROSS-CUTTING]` tag from v0.3.0 is inherited and requires full MOD decomposition. The `[EXTERNAL]` tag is the single canonical tag for third-party library wrappers — `[COTS]` is not recognized. All requirements are atomized from the feature specification using the strict translator constraint.

## Requirements

### Functional Requirements

| ID | Description | Priority | Rationale | Verification Method |
|----|-------------|----------|-----------|---------------------|
| REQ-001 | The extension SHALL provide a `/speckit.v-model.module-design` command that reads `architecture-design.md` as input and produces `module-design.md` as output containing `MOD-NNN` identifiers organized into four mandatory views. | P1 | US1: Module design is the foundational artifact of the bottom V-level; without it, unit tests cannot be generated and Matrix D cannot be built. | Test |
| REQ-002 | The `/speckit.v-model.module-design` command SHALL assign each module a unique `MOD-NNN` identifier (3-digit zero-padded, e.g., `MOD-001`, `MOD-012`), sequentially numbered and never renumbered once assigned. | P1 | US1, FR-021: Traceable identifiers are required for deterministic coverage validation and traceability matrix generation. | Inspection |
| REQ-003 | The `module-design.md` output SHALL include an **Algorithmic / Logic View** containing step-by-step pseudocode enclosed in fenced Markdown code blocks tagged `pseudocode` (i.e., ` ```pseudocode ... ``` `) defining exactly how each module transforms inputs into outputs, with every branch, loop, and decision point explicit. | P1 | US1, FR-002, FR-009, DO-178C: Low-Level Requirements must be detailed enough that coding is merely a translation exercise. Fenced pseudocode blocks enable deterministic structural validation (presence check), while semantic quality is evaluated by the LLM-as-judge suite (SC-007). | Inspection |
| REQ-004 | Each `MOD-NNN` item SHALL include a **Target Source File(s)** property mapping the module to one or more physical file paths in the repository (e.g., `src/sensor/parser.py` or `src/sensor/parser.h, src/sensor/parser.cpp`), with comma-separated paths supported for languages where a single module spans multiple files (header/implementation pairs). | P1 | US1, FR-003: The Source File mapping bridges the gap between specification and codebase, enabling `/speckit.implement` to know exactly where to write code. Comma-separated paths handle C/C++ header/impl pairs without forcing a single-file choice. | Inspection |
| REQ-005 | The `module-design.md` output SHALL include a **State Machine View** using Mermaid `stateDiagram-v2` syntax for stateful modules. | P1 | US1, FR-002: Stateful modules require explicit state/transition/event/guard documentation for State Transition Testing. | Inspection |
| REQ-006 | Stateless modules SHALL be marked with a bypass string in the State Machine View that is detectable by a broad, case-insensitive regex pattern (e.g., `(?i)N/?A.*Stateless`). Validation scripts SHALL use this broad pattern rather than exact string matching to accommodate minor LLM punctuation variance (em-dash vs. hyphen, casing differences). | P1 | FR-002, Red Team: LLMs produce punctuation variance; broad regex prevents brittle validation failures. | Test |
| REQ-007 | The `module-design.md` output SHALL include an **Internal Data Structures** section documenting local variables, constants, buffers, and object classes with explicit types, sizes, and constraints (e.g., "buffer is exactly 256 bytes, uint8_t array, initialized to 0x00"). | P1 | US1, FR-002: Explicit data structure constraints feed Boundary Value Analysis at the variable level. | Inspection |
| REQ-008 | The `module-design.md` output SHALL include an **Error Handling & Return Codes** section documenting exactly how each module catches exceptions internally and formats them to honor the contract defined in the Architecture Interface View. | P1 | US1, FR-002: Error handling must trace back to the ARCH interface contract to maintain V-Model consistency. | Inspection |
| REQ-009 | Each `MOD-NNN` item SHALL trace to one or more `ARCH-NNN` identifiers via an explicit "Parent Architecture Modules" field. | P1 | US1, FR-004: Forward traceability from architecture modules to module designs is required for deterministic coverage validation. | Test |
| REQ-010 | The module design command SHALL support many-to-many ARCH↔MOD relationships: a single `ARCH-NNN` MAY map to multiple `MOD-NNN` items, and a single `MOD-NNN` MAY satisfy multiple `ARCH-NNN` modules. | P1 | US1, FR-005: Real systems have modules that serve multiple architectural concerns. | Test |
| REQ-011 | `ARCH-NNN` modules tagged `[CROSS-CUTTING]` in the architecture design SHALL be fully decomposed into `MOD-NNN` items with complete pseudocode, inheriting the `[CROSS-CUTTING]` tag. Infrastructure code (loggers, error handlers, thread pools) requires the same rigor as business logic. | P1 | US1 Scenario 2, FR-006: Cross-cutting modules have implementation logic that requires unit testing. | Test |
| REQ-012 | If an `ARCH-NNN` module represents a third-party library wrapper, the module design command SHALL tag the resulting `MOD-NNN` as `[EXTERNAL]` and document only the wrapper/configuration interface (connection parameters, retry policy, timeout config), omitting deep algorithmic pseudocode of the library internals. | P1 | US1 Scenario 3, FR-007: Prevents LLM hallucination of third-party library internals. | Test |
| REQ-013 | When an `[EXTERNAL]` module's wrapper contains meaningful logic (e.g., retry policy, circuit breaker, connection pooling), that wrapper logic SHALL be documented with pseudocode and unit-tested. Only the third-party library's internal algorithm SHALL be bypassed. | P1 | Edge case 7: The `[EXTERNAL]` tag applies to the library, not the wrapper. | Test |
| REQ-014 | When the project is configured for a safety-critical domain (DO-178C Level A, ISO 26262, IEC 62304), the module design output SHALL include additional sections: **Complexity Constraints** (cyclomatic complexity ≤ 10 per function, MISRA C/CERT-C rule annotations per module), **Memory Management** (no dynamic allocation after initialization, no unbounded loops), and **Single Entry/Exit** (one return point per function for deterministic execution paths). | P1 | US1 Scenario 4, FR-008, DO-178C/ISO 26262/MISRA C:2012: Safety-critical module design requires explicit coding constraints. | Test |
| REQ-015 | When the module design process identifies a necessary function or class not traceable to any `ARCH-NNN`, the command SHALL flag it as `[DERIVED MODULE: description]` rather than silently creating a `MOD-NNN`, prompting the user to update `architecture-design.md`. | P1 | FR-010, Edge case 6: Derived modules must flow back up the V-Model for proper traceability. | Test |
| REQ-016 | The module design Algorithmic/Logic View SHALL NOT contain vague or descriptive prose in place of pseudocode — every `MOD-NNN` (except `[EXTERNAL]`) SHALL contain a fenced ` ```pseudocode ``` ` code block. The absence of this block SHALL be flagged as a structural failure by deterministic validators. | P1 | FR-009, SC-007: Pseudocode concreteness is the key quality differentiator at the module level. Structural presence is checked deterministically; semantic quality is evaluated by LLM-as-judge. | Test |
| REQ-017 | The extension SHALL provide a `/speckit.v-model.unit-test` command that reads `module-design.md` as input and produces `unit-test.md` as output containing `UTP-NNN-X` test cases and `UTS-NNN-X#` test scenarios. | P1 | US2: Unit testing is the right side of the V at the lowest level, paired with module design (Constitution P1). | Test |
| REQ-018 | The `/speckit.v-model.unit-test` command SHALL assign test case identifiers using the format `UTP-NNN-X` (e.g., `UTP-001-A`) where NNN matches the parent `MOD-NNN` and X is a sequential uppercase letter. | P1 | US2, FR-022: ID lineage encoding enables deterministic ancestry tracing from any test case to its parent module. | Test |
| REQ-019 | The `/speckit.v-model.unit-test` command SHALL assign test scenario identifiers using the format `UTS-NNN-X#` (e.g., `UTS-001-A1`) where NNN-X matches the parent `UTP-NNN-X` and # is a sequential number. | P1 | US2, FR-023: Three-tier ID schema (MOD → UTP → UTS) mirrors the architecture tier (ARCH → ITP → ITS). | Test |
| REQ-020 | Each unit test case (`UTP-NNN-X`) SHALL apply one of the mandatory ISO 29119-4 white-box techniques: **Statement & Branch Coverage** (execute every line and every True/False branch), **Boundary Value Analysis** (at the variable level: min, min-1, mid, max, max+1 for scalar ordered types), **Equivalence Partitioning** (for discrete non-scalar types — Booleans, Enums — where numeric boundaries do not logically exist), **Strict Isolation** (every external dependency mocked), or **State Transition Testing** (every state transition including invalid ones). | P1 | US2, FR-012, Red Team: BVA applies to scalar ordered types; Equivalence Partitioning is the correct technique for discrete types like Booleans and Enums. | Inspection |
| REQ-021 | Each unit test case (`UTP-NNN-X`) SHALL explicitly name its technique and reference the specific module view that drives it (Algorithmic/Logic View for Statement & Branch Coverage, Internal Data Structures for BVA/EP, Architecture Interface View for Strict Isolation, State Machine View for State Transition Testing). | P1 | US2, FR-013: Anchoring tests to views and techniques prevents generic "verify it works" tests. | Inspection |
| REQ-022 | Each unit test case (`UTP-NNN-X`) SHALL include a **Dependency & Mock Registry** table explicitly listing all external dependencies the module has (from Architecture Interface View) and exactly how each is mocked/stubbed (e.g., `Mock: DatabaseConnection → Returns static JSON`). | P1 | US2, FR-014: Prevents accidental integration in unit tests and provides auditors with explicit isolation evidence. | Inspection |
| REQ-023 | The Dependency & Mock Registry SHALL explicitly include hardware interfaces (e.g., GPIO pins, memory-mapped registers, I2C/SPI buses) as dependencies that require mocking/stubbing — LLMs often fail to recognize hardware registers as "dependencies" in embedded contexts. | P1 | FR-014, Red Team: Hardware interfaces in automotive/medical modules are external dependencies that must be isolated during unit testing. | Inspection |
| REQ-024 | When a `MOD-NNN` has no external dependencies, the Dependency & Mock Registry SHALL show "None — module is self-contained" and the unit test SHALL proceed with direct invocation (no mocking needed). | P1 | Edge case 5: Self-contained modules are valid and must not fail validation due to an empty mock registry. | Inspection |
| REQ-025 | Unit test scenarios (`UTS-NNN-X#`) SHALL use Arrange/Act/Assert structure with **white-box, implementation-oriented language** referencing internal code paths, variables, and branches (e.g., "Arrange: Set buffer_size to 256 and input_length to 257, Act: Call parse_sensor_data(), Assert: Returns ERROR_OVERFLOW and buffer remains unchanged"), distinct from black-box acceptance or integration-level boundary language. | P1 | US2, FR-015: White-box language is fundamentally different from the Given/When/Then BDD structure used at higher V-Model levels. | Inspection |
| REQ-026 | The unit test command SHALL NOT generate unit test cases for `MOD-NNN` items tagged `[EXTERNAL]`. Each skipped module SHALL be noted: "Module MOD-NNN is [EXTERNAL] — wrapper behavior tested at integration level." | P1 | US2 Scenario 4, FR-016: Unit testing third-party library internals generates meaningless, hallucinated test cases. | Test |
| REQ-027 | When the project is configured for a safety-critical domain (DO-178C Level A, ISO 26262 ASIL-D), the unit test output SHALL include additional techniques: **MC/DC (Modified Condition/Decision Coverage)** with explicit boolean truth tables for complex conditional branches proving each individual condition can independently affect the decision outcome, and **Variable-Level Fault Injection** forcing local variables into corrupted states to verify internal error handling. | P1 | US2 Scenario 2, FR-017, DO-178C Level A: MC/DC is the required structural coverage metric for safety-critical unit testing. | Test |
| REQ-028 | The unit test command SHALL invoke `validate-module-coverage.sh` as a post-generation coverage gate and include the validation result (pass/fail with coverage summary) in its output. | P1 | US2, FR-018: The coverage gate ensures no module is left untested. | Test |
| REQ-029 | The unit test command SHALL NOT generate tests for user-journey scenarios, integration-level module boundaries, or system-level behavior — only tests that verify internal module logic, branches, data transformations, and state transitions. | P1 | FR-019: Higher-level testing is the responsibility of acceptance, system, and integration test layers. | Inspection |
| REQ-030 | For stateful modules with a State Machine View, the State Transition Testing technique SHALL cover every defined transition (including invalid transitions that should be rejected) as defined in the module design. | P1 | US2 Scenario 5, FR-012: Invalid transitions are a critical source of defects in stateful code. | Test |
| REQ-031 | The ID lineage encoding SHALL be consistent within the module tier: given any `UTS-NNN-X#` identifier, a regex SHALL be able to extract the parent `UTP-NNN-X` and grandparent `MOD-NNN` without consulting any lookup table. | P1 | SC-003, FR-020: Machine-parseable lineage is required for deterministic script validation. | Test |
| REQ-032 | To resolve the `ARCH-NNN` ancestor(s) from a `MOD-NNN` identifier, the script SHALL consult the "Parent Architecture Modules" field in `module-design.md`, because many-to-many ARCH↔MOD relationships (REQ-010) make it mathematically impossible to derive the ARCH parent from the ID string alone. | P1 | FR-020, Red Team: The regex-to-full-ancestry claim was a known hallucination caught in v0.3.0 and again in v0.4.0 specification. | Test |
| REQ-033 | The extension SHALL provide a `validate-module-coverage.sh` Bash script that deterministically validates forward coverage: every `ARCH-NNN` (including `[CROSS-CUTTING]`) in `architecture-design.md` has at least one corresponding `MOD-NNN` in `module-design.md`. | P1 | US3, FR-024: Deterministic verification (Constitution P2) requires script-based, not AI-based, coverage validation. | Test |
| REQ-034 | The `validate-module-coverage.sh` script SHALL deterministically validate backward coverage: every `MOD-NNN` (except `[EXTERNAL]`) in `module-design.md` has at least one corresponding `UTP-NNN-X` in `unit-test.md`. | P1 | US3, FR-024: Backward coverage proves no module was designed without a paired unit test. | Test |
| REQ-035 | The `validate-module-coverage.sh` script SHALL correctly bypass the `UTP` requirement for `MOD-NNN` items tagged `[EXTERNAL]`, counting them as "covered by integration tests" in the summary rather than flagging them as gaps. | P1 | US3 Scenario 4, FR-025: External library wrappers are tested at the integration level, not unit level. | Test |
| REQ-036 | The `validate-module-coverage.sh` script SHALL support partial validation: if `unit-test.md` is not present, it SHALL validate only the ARCH → MOD forward coverage leg and gracefully skip the MOD → UTP backward checks, exiting with code 0 if forward coverage is complete. | P1 | US3 Scenario 5, FR-024b: Enables CI to run after `/speckit.v-model.module-design` before `/speckit.v-model.unit-test` has been executed. | Test |
| REQ-037 | The `validate-module-coverage.sh` script SHALL detect and report orphaned identifiers: any `MOD-NNN` whose parent `ARCH-NNN` does not exist in `architecture-design.md`, and any `UTP-NNN-X` whose parent `MOD-NNN` does not exist in `module-design.md`. | P1 | US3 Scenario 3, FR-024: Orphaned artifacts indicate a broken traceability chain. | Test |
| REQ-038 | The `validate-module-coverage.sh` script SHALL exit with code 0 when all applicable coverage checks pass and code 1 when any gap or orphan is detected. | P1 | US3, FR-026: Non-zero exit codes enable CI hard-fail enforcement (Constitution P2 quality gate). | Test |
| REQ-039 | The `validate-module-coverage.sh` script SHALL output human-readable gap reports listing each specific gap or orphan by ID (e.g., "ARCH-003: no module design mapping found") suitable for CI log inspection. | P1 | US3 Scenario 2, FR-026: Developers must be able to identify and fix specific gaps from the CI output. | Test |
| REQ-040 | The `validate-module-coverage.sh` script SHALL support `--json` output mode for programmatic consumption by the unit test command. | P1 | FR-027: Machine-readable output enables the unit test command to parse coverage results. | Test |
| REQ-041 | The `/speckit.v-model.module-design` command SHALL follow the strict translator constraint: when deriving from `architecture-design.md`, the AI SHALL NOT invent, infer, or add modules for capabilities not present in the architecture design. | P1 | FR-035, Constitution P5: Prevents AI hallucination of implementation features. | Test |
| REQ-042 | The `/speckit.v-model.unit-test` command SHALL follow the strict translator constraint: when deriving from `module-design.md`, the AI SHALL NOT invent test cases for modules or logic not present in the module design. | P1 | FR-035, Constitution P5: Prevents AI hallucination of test scenarios. | Test |
| REQ-043 | The module design command SHALL fail gracefully with a clear error message ("No architecture modules found in architecture-design.md — cannot generate module design") when `architecture-design.md` is empty or contains zero `ARCH-NNN` identifiers. | P1 | Edge case 1: Empty input must not produce a malformed output. | Test |

## Assumptions

- Users already have a valid `architecture-design.md` generated by `/speckit.v-model.architecture-design` (v0.3.0). The module design command does not generate architecture designs — it decomposes architecture modules into implementation-level specifications.
- The many-to-many ARCH↔MOD relationship is documented explicitly in the module design's "Parent Architecture Modules" field, not inferred at runtime. Each `MOD-NNN` entry explicitly lists its parent `ARCH-NNN` IDs.
- The `validate-module-coverage.sh` script uses regex-based parsing consistent with `validate-architecture-coverage.sh` from v0.3.0 — no runtime database or external tooling required.
- Backward compatibility with v0.3.0 artifacts is mandatory — existing files are never modified by v0.4.0 operations.
- Safety-critical sections are optional, activated by domain configuration in `v-model-config.yml`. Projects not targeting regulated domains get the four mandatory views/techniques without the safety additions.
- The `[CROSS-CUTTING]` tag is inherited from v0.3.0 architecture modules — cross-cutting MODs still require full decomposition and unit testing.
- The `[EXTERNAL]` tag applies to the third-party library internals, not the wrapper. If a wrapper contains meaningful logic (retry policy, circuit breaker), that logic IS documented and unit-tested. Only the library's internal algorithm is bypassed.
- The unit test command generates specification-level test cases, not executable test code. The generated `UTS-NNN-X#` scenarios are designed to be consumed by `/speckit.implement`, which writes the actual test code (TDD: tests first, then implementation).

## Glossary

| Term | Definition |
|------|-----------|
| DO-178C | Software Considerations in Airborne Systems and Equipment Certification. Defines Low-Level Requirements (LLR) that must be detailed enough for mechanical code translation. |
| ISO 26262 Part 6 | Automotive functional safety — Product development at the software level. Mandates restricted complexity, MISRA C compliance, and variable-level fault injection. |
| ISO/IEC/IEEE 29119-4 | Software Testing — Test Techniques. Defines White-Box techniques (Statement/Branch Coverage, BVA, Equivalence Partitioning, State Transition Testing) used at the unit level. |
| MISRA C:2012 | Motor Industry Software Reliability Association — safe coding standard constraining complexity, memory management, and control flow. Cyclomatic complexity ≤ 10 per function. |
| CERT-C | SEI CERT C Coding Standard — secure coding rules complementing MISRA for high-integrity software. |
| MC/DC | Modified Condition/Decision Coverage. The most rigorous structural coverage metric — proves each individual boolean condition can independently affect the decision outcome. Required by DO-178C Level A. |
| MOD-NNN | Module design identifier (e.g., MOD-001). 3-digit zero-padded. Maps many-to-many with ARCH-NNN. Each includes Target Source File(s) for codebase mapping. |
| UTP-NNN-X | Unit test case identifier (e.g., UTP-001-A). Parent: MOD-NNN. Each names one of the mandatory white-box techniques and includes a Dependency & Mock Registry. |
| UTS-NNN-X# | Unit test scenario identifier (e.g., UTS-001-A1). Parent: UTP-NNN-X. Arrange/Act/Assert format with white-box, implementation-oriented language. |
| Dependency & Mock Registry | Table within each UTP listing all external dependencies (including hardware interfaces like GPIO, memory-mapped registers, I2C/SPI) and their mock/stub strategy. Prevents accidental integration in unit tests. |
| Target Source File(s) | Property mapping a MOD-NNN to one or more physical repository file paths. Comma-separated for header/implementation pairs (e.g., `parser.h, parser.cpp`). Bridges specification to code. |
| Statement & Branch Coverage | White-box technique executing every line and every True/False outcome of every conditional. Driven by Algorithmic/Logic View. |
| Boundary Value Analysis (BVA) | White-box technique testing at variable boundaries: min, min-1, mid, max, max+1. Applies to scalar ordered types. Driven by Internal Data Structures. |
| Equivalence Partitioning (EP) | White-box technique for discrete non-scalar types (Booleans, Enums) where numeric boundaries do not exist. Fallback for BVA when boundaries are not applicable. |
| Strict Isolation | Testing principle that unit tests must never hit real databases, networks, or hardware. All external dependencies are mocked/stubbed per the Dependency & Mock Registry. |
| State Transition Testing | White-box technique covering every state transition (including invalid ones) defined in the State Machine View. |
| Variable-Level Fault Injection | Safety-critical technique forcing local variables into corrupted states to verify internal error handling. Required by ISO 26262 ASIL-D. |
| `[CROSS-CUTTING]` | Tag inherited from v0.3.0 for infrastructure/utility modules (loggers, error handlers, thread pools) that serve the system as a whole. Requires full MOD decomposition and UTP coverage. |
| `[EXTERNAL]` | Single canonical tag for third-party library wrapper modules. MOD documents wrapper interface only; UTP coverage is bypassed (tested at integration level). `[COTS]` is not recognized. |
| `[DERIVED MODULE]` | Flag for modules identified during design that are not traceable to any parent architecture module. Must be flagged, not silently added. |
| Matrix A (Validation) | Traceability matrix: REQ → ATP → SCN. Proves the system does what the user needs. |
| Matrix B (System Verification) | Traceability matrix: REQ → SYS → STP → STS. Proves the system architecture works as designed. |
| Matrix C (Integration Verification) | Traceability matrix: SYS → ARCH → ITP → ITS. Proves the module boundaries work correctly. |
| Matrix D (Implementation Verification) | Traceability matrix: ARCH → MOD → UTP → UTS. Proves individual modules are correctly implemented and tested. Completes the full V-Model chain. Requires three input files: module-design.md, unit-test.md, and architecture-design.md (for ARCH → SYS lineage). |
