# V-Model Requirements Specification: Architecture Design ↔ Integration Testing

**Feature Branch**: `003-architecture-integration`  
**Created**: 2026-02-21  
**Status**: Approved  
**Source**: `specs/003-architecture-integration/spec.md`

## Overview

This document formalizes the requirements for extending the V-Model Extension Pack from System Design ↔ System Testing (v0.2.0) down to Architecture Design ↔ Integration Testing (v0.3.0). The extension adds two new commands (`/speckit.v-model.architecture-design` and `/speckit.v-model.integration-test`), a deterministic validation script, extended traceability matrix support, and corresponding templates. Architecture design follows ISO/IEC/IEEE 42010 and Kruchten's 4+1 View Model with four mandatory views (Logical, Process, Interface, Data Flow). Integration testing follows ISO/IEC/IEEE 29119-4 with four mandatory techniques (Interface Contract Testing, Data Flow Testing, Interface Fault Injection, Concurrency & Race Condition Testing). Safety-critical additions (ASIL Decomposition, Defensive Programming, Temporal Constraints, SIL/HIL Compatibility, Resource Contention) are conditional on domain configuration. All requirements are atomized from the feature specification using the strict translator constraint.

## Requirements

### Functional Requirements

| ID | Description | Priority | Rationale | Verification Method |
|----|-------------|----------|-----------|---------------------|
| REQ-001 | The extension SHALL provide a `/speckit.v-model.architecture-design` command that reads `system-design.md` as input and produces `architecture-design.md` as output. | P1 | US1: Architecture design decomposition is the foundational artifact of this V-level; without it, integration tests cannot be generated and Matrix C cannot be built. | Test |
| REQ-002 | The `/speckit.v-model.architecture-design` command SHALL assign each architecture module a unique `ARCH-NNN` identifier (e.g., `ARCH-001`, `ARCH-012`), 3-digit zero-padded, sequentially numbered and never renumbered once assigned. | P1 | US1, FR-016: Traceable identifiers are required for deterministic coverage validation and traceability matrix generation. | Inspection |
| REQ-003 | The `architecture-design.md` output SHALL include a **Logical View (Component Breakdown)** listing how each `SYS-NNN` is decomposed into software modules (`ARCH-NNN`), with each module's name, description, and list of parent `SYS-NNN` identifiers via an explicit "Parent System Components" field. | P1 | US1, FR-002, ISO/IEC/IEEE 42010: The Logical View proves each system component is decomposed into implementable modules with traceable ancestry. | Inspection |
| REQ-004 | The `architecture-design.md` output SHALL include a **Process View (Dynamic Behavior)** documenting runtime module interactions — sequences, concurrency, thread management — with Mermaid sequence diagrams embedded in Markdown showing execution order, synchronization points, and thread/task boundaries. | P1 | US1, FR-002, Kruchten 4+1: The Process View shows how modules interact at runtime, which directly drives Concurrency & Race Condition Testing. | Inspection |
| REQ-005 | The `architecture-design.md` output SHALL include an **Interface View (Strict API / Interface Contracts)** documenting for every `ARCH-NNN` module: inputs accepted (types, formats, ranges), outputs produced (types, guarantees), and exceptions thrown (error codes, failure modes). | P1 | US1, FR-002, FR-006: The Interface View defines strict contracts that directly drive Interface Contract Testing and Interface Fault Injection. | Inspection |
| REQ-006 | The `architecture-design.md` output SHALL include a **Data Flow View (Data Control Flow)** tracing how data moves through modules — input → transformation chain → output with intermediate formats. | P1 | US1, FR-002: The Data Flow View feeds Data Flow Testing technique by documenting the expected transformation chain. | Inspection |
| REQ-007 | Each business-logic `ARCH-NNN` module SHALL trace to one or more `SYS-NNN` identifiers via the "Parent System Components" field in the Logical View. | P1 | US1, FR-003: Forward traceability from system components to architecture modules is required for deterministic coverage validation. | Test |
| REQ-008 | The architecture design command SHALL support many-to-many SYS↔ARCH relationships: a single `SYS-NNN` MAY map to multiple `ARCH-NNN` modules, and a single `ARCH-NNN` module MAY satisfy multiple `SYS-NNN` components. | P1 | US1, FR-004, Scenario 3: Real systems have cross-cutting requirements that span multiple subsystems. | Test |
| REQ-009 | `ARCH-NNN` modules that represent infrastructure or utility concerns (e.g., logging, secrets management, thread pools, error handling frameworks) SHALL trace to a `[CROSS-CUTTING]` tag instead of a `SYS-NNN` identifier. | P1 | FR-007, Red Team Critique 1: Utility modules are architecturally necessary but do not trace to a single system component — they serve the system as a whole. | Inspection |
| REQ-010 | The Logical View SHALL clearly distinguish business-logic modules (with explicit SYS parents) from cross-cutting modules (with the `[CROSS-CUTTING]` tag and a rationale explaining why the module is system-wide). | P1 | FR-007: Auditors need to see the distinction between business logic and infrastructure at a glance. | Inspection |
| REQ-011 | Cross-cutting modules (`[CROSS-CUTTING]` tag) SHALL still be included in coverage validation: each cross-cutting `ARCH-NNN` MUST have at least one `ITP-NNN-X` integration test case. | P1 | FR-007: Infrastructure modules have integration boundaries that must be tested. | Test |
| REQ-012 | When the architecture design process identifies a necessary technical module that is neither traceable to any `SYS-NNN` nor qualifies as `[CROSS-CUTTING]`, the command SHALL flag it as `[DERIVED MODULE: description]` rather than silently creating an `ARCH-NNN`, prompting the user to update `system-design.md`. | P1 | FR-007a, Edge case 9, DO-178C/ISO 26262: Derived modules must flow back up the V-Model for proper traceability. | Test |
| REQ-013 | The architecture design command SHALL reject "black box" descriptions — every `ARCH-NNN` module SHALL have a complete interface contract in the Interface View (inputs, outputs, exceptions). Modules with incomplete contracts SHALL trigger an anti-pattern warning. | P1 | FR-006, Edge case 8: Incomplete contracts produce untestable modules. | Test |
| REQ-014 | The extension SHALL provide a `/speckit.v-model.integration-test` command that reads `architecture-design.md` as input and produces `integration-test.md` as output. | P1 | US2: Integration testing is the right side of the V at the architecture level, paired with architecture design (Constitution P1). | Test |
| REQ-015 | The `/speckit.v-model.integration-test` command SHALL assign test case identifiers using the format `ITP-NNN-X` (e.g., `ITP-001-A`) where NNN matches the parent `ARCH-NNN` and X is a sequential uppercase letter. | P1 | US2, FR-017: ID lineage encoding enables deterministic ancestry tracing from any test case to its parent module. | Test |
| REQ-016 | The `/speckit.v-model.integration-test` command SHALL assign test scenario identifiers using the format `ITS-NNN-X#` (e.g., `ITS-001-A1`) where NNN-X matches the parent `ITP-NNN-X` and # is a sequential number. | P1 | US2, FR-018: Three-tier ID schema (ARCH → ITP → ITS) mirrors the system tier (SYS → STP → STS). | Test |
| REQ-017 | Each integration test case (`ITP-NNN-X`) SHALL apply one of the four mandatory ISO 29119-4 techniques: Interface Contract Testing (driven by Interface View), Data Flow Testing (driven by Data Flow View), Interface Fault Injection (driven by Interface View error contracts + Process View timing), or Concurrency & Race Condition Testing (driven by Process View concurrency model). | P1 | US2, FR-009: Named techniques force rigorous, auditor-grade integration test design. | Inspection |
| REQ-018 | Each integration test case (`ITP-NNN-X`) SHALL explicitly name its technique and reference the specific architecture view that drives it. | P1 | US2, FR-010: Anchoring tests to views and techniques prevents generic "verify it works" tests. | Inspection |
| REQ-019 | Integration test scenarios (`ITS-NNN-X#`) SHALL use Given/When/Then BDD structure with **technical, module-boundary-oriented language** (e.g., "Given Module A sends a malformed payload to Module B, When Module B receives the payload, Then Module B rejects it with error code INVALID_SCHEMA and does not propagate the error downstream"), distinct from user-centric acceptance language and system-level test language. | P1 | US2, FR-011: Integration tests verify module boundaries, not user journeys or internal logic. | Inspection |
| REQ-020 | The integration test command SHALL NOT generate tests for internal module logic or user-journey scenarios — only tests that verify the boundaries and handshakes between modules. | P1 | FR-014: Internal logic is unit test responsibility (v0.4.0); user journeys are acceptance test responsibility. | Inspection |
| REQ-021 | When the project is configured for a safety-critical domain (ISO 26262, DO-178C, IEC 62304), the architecture design output SHALL include additional sections: **ASIL Decomposition** (safety integrity allocation per module, e.g., SYS ASIL-D → ARCH-001 ASIL-D + ARCH-002 ASIL-A), **Defensive Programming** (how invalid inputs from one module are caught before corrupting the next), and **Temporal & Execution Constraints** (watchdog timers, execution order, deadlock prevention). | P1 | US1 Scenario 4, FR-005, ISO 26262/DO-178C: Safety-critical decomposition requires explicit integrity allocation and defensive measures. | Test |
| REQ-022 | When the project is configured for a safety-critical domain, the integration test output SHALL include additional sections: **SIL/HIL Compatibility** (scenarios executable in Software/Hardware-in-the-Loop environments) and **Resource Contention** (proving modules don't exhaust shared resources during interaction). | P1 | US2 Scenario 4, FR-012, ISO 26262/DO-178C: Safety-critical integration testing must address hardware-loop execution and resource exhaustion. | Test |
| REQ-023 | The integration test command SHALL invoke `validate-architecture-coverage.sh` as a post-generation coverage gate and include the validation result (pass/fail with coverage summary) in its output. | P1 | US2, FR-013: The coverage gate ensures no architecture module is left untested. | Test |
| REQ-024 | The extension SHALL provide a `validate-architecture-coverage.sh` Bash script that deterministically validates forward coverage: every `SYS-NNN` in `system-design.md` has at least one corresponding `ARCH-NNN` in `architecture-design.md`. | P1 | US3, FR-019: Deterministic verification (Constitution P2) requires script-based, not AI-based, coverage validation. | Test |
| REQ-025 | The `validate-architecture-coverage.sh` script SHALL deterministically validate backward coverage: every `ARCH-NNN` in `architecture-design.md` has at least one corresponding `ITP-NNN-X` in `integration-test.md`. | P1 | US3, FR-019: Backward coverage proves no module was designed without a paired integration test. | Test |
| REQ-026 | The `validate-architecture-coverage.sh` script SHALL detect and report orphaned identifiers: any `ARCH-NNN` not referenced as a parent in any `SYS-NNN` or `[CROSS-CUTTING]` tag, and any `ITP-NNN-X` whose parent `ARCH-NNN` does not exist in `architecture-design.md`. | P1 | US3, FR-019, Scenario 3: Orphaned artifacts indicate a broken traceability chain. | Test |
| REQ-027 | The `validate-architecture-coverage.sh` script SHALL exit with code 0 when all coverage checks pass and code 1 when any gap or orphan is detected. | P1 | US3, FR-020: Non-zero exit codes enable CI hard-fail enforcement (Constitution P2 quality gate). | Test |
| REQ-028 | The `validate-architecture-coverage.sh` script SHALL output human-readable gap reports listing each specific gap or orphan by ID (e.g., "SYS-003: no architecture module mapping found") suitable for CI log inspection. | P1 | US3, FR-020, Scenarios 2-3: Developers must be able to identify and fix specific gaps from the CI output. | Test |
| REQ-029 | The `validate-architecture-coverage.sh` script SHALL support `--json` output mode for programmatic consumption by the integration test command. | P1 | FR-021: Machine-readable output enables the integration test command to parse coverage results. | Test |
| REQ-030 | The ID lineage encoding SHALL be consistent within the architecture tier: given any `ITS-NNN-X#` identifier, a regex SHALL be able to extract the parent `ITP-NNN-X` and grandparent `ARCH-NNN` without consulting any lookup table. To resolve the `SYS-NNN` ancestor(s), the script SHALL consult the "Parent System Components" field in `architecture-design.md`, because the many-to-many SYS↔ARCH relationship (REQ-008) makes pure string-based SYS extraction impossible. | P1 | SC-003, FR-015: Machine-parseable lineage is required for deterministic script validation; SYS resolution requires file lookup due to many-to-many mapping. | Test |
| REQ-031 | The `/speckit.v-model.trace` command SHALL be extended to produce **Matrix C (Integration Verification)** — `SYS → ARCH → ITP → ITS` — when `architecture-design.md` and `integration-test.md` exist in the feature directory. `ARCH-NNN` modules tagged as `[CROSS-CUTTING]` SHALL appear in Matrix C as pseudo-rows where the `SYS` column displays `N/A (Cross-Cutting)` to ensure infrastructure modules are not excluded from the audit artifact. | P2 | US4, FR-023: The traceability matrix is the key audit artifact; cross-cutting modules must not fall through the cracks. | Test |

| REQ-035 | The `/speckit.v-model.trace` command SHALL build matrices progressively: Matrix A alone after acceptance, A+B after system-test, A+B+C after integration-test. | P2 | FR-025: Progressive matrix generation matches the progressive V-Model descent. | Test |
| REQ-036 | Each matrix SHALL include an independently calculated coverage percentage that matches the output of the corresponding deterministic validation script. | P2 | US4, SC-004: Coverage percentages must be deterministically verified, not AI-estimated. | Test |

## Assumptions

- Users already have a valid `system-design.md` generated by `/speckit.v-model.system-design` (v0.2.0). The architecture design command does not generate system designs — it decomposes system components into software modules.
- The many-to-many SYS↔ARCH relationship is documented explicitly in the Logical View's "Parent System Components" field, not inferred at runtime. Each `ARCH-NNN` entry explicitly lists its parent `SYS-NNN` IDs or a `[CROSS-CUTTING]` tag.
- The `validate-architecture-coverage.sh` script uses regex-based parsing consistent with `validate-system-coverage.sh` from v0.2.0 — no runtime database or external tooling required.
- Backward compatibility with v0.2.0 artifacts is mandatory — existing files are never modified by v0.3.0 operations.
- Safety-critical sections are optional, activated by domain configuration in `v-model-config.yml`.
- Cross-cutting modules (logging, secrets management, thread pools) are architecturally necessary and do not trace to individual system components — they are tagged `[CROSS-CUTTING]` rather than forced into artificial SYS mappings.

## Dependencies

- **Spec Kit ≥ 0.1.0**: Required platform version for extension command registration and agent system.
- **V-Model Extension v0.2.0**: Existing `system-design.md`, `system-test.md`, `validate-system-coverage.sh`, `build-matrix.sh`, and `trace.md` command must be present and functional.
- **Bash ≥ 4.0**: Required for `validate-architecture-coverage.sh` (regex, associative arrays).
- **PowerShell ≥ 7.0**: Required for `validate-architecture-coverage.ps1` cross-platform parity.

## Glossary

| Term | Definition |
|------|-----------|
| ISO/IEC/IEEE 42010 | Systems and Software Engineering — Architecture Description. Defines the framework for documenting architecture viewpoints and views. |
| Kruchten's 4+1 View Model | Defines four mandatory architecture views (Logical, Process, Interface, Data Flow) plus scenarios that link them. |
| ISO/IEC/IEEE 29119-4 | Software Testing — Test Techniques. Defines integration test techniques (Interface Contract Testing, Data Flow Testing, Fault Injection). |
| Logical View | 42010/4+1 view showing how system components are decomposed into software modules (SYS → ARCH breakdown). |
| Process View | 42010/4+1 view showing runtime module interactions, concurrency, and thread management via Mermaid sequence diagrams. |
| Interface View | 42010/4+1 view defining strict API contracts per module — inputs, outputs, exceptions. |
| Data Flow View | 42010/4+1 view tracing data transformation chains — input → intermediate formats → output. |
| ARCH-NNN | Architecture module identifier (e.g., ARCH-001). Maps many-to-many with SYS-NNN, or tagged `[CROSS-CUTTING]`. |
| ITP-NNN-X | Integration test case identifier (e.g., ITP-001-A). Parent: ARCH-NNN. Each names one of four ISO 29119-4 techniques. |
| ITS-NNN-X# | Integration test scenario identifier (e.g., ITS-001-A1). Parent: ITP-NNN-X. BDD format with module-boundary language. |
| Interface Contract Testing | ISO 29119-4 technique verifying that a provider module honors the data contract expected by a consumer module. Driven by Interface View. |
| Data Flow Testing | ISO 29119-4 technique verifying end-to-end data transformation chains across module boundaries. Driven by Data Flow View. |
| Interface Fault Injection | ISO 29119-4 technique injecting malformed payloads, timeouts, and error conditions at module boundaries. Driven by Interface View + Process View. |
| Concurrency & Race Condition Testing | ISO 29119-4 technique verifying mutex strategies, lock handling, and queue ordering under simultaneous access. Driven by Process View. |
| CROSS-CUTTING | Tag applied to infrastructure/utility ARCH modules (logging, secrets, thread pools) that serve the system as a whole rather than tracing to a specific SYS component. |
| Derived Module | A technical module identified during architecture design that is neither traceable to a SYS component nor qualifies as CROSS-CUTTING. Must be flagged, not silently added. |
| ASIL Decomposition | ISO 26262 concept — allocation of safety integrity levels from system components to architecture modules. |
| FFI | Freedom from Interference — ISO 26262 concept proving isolation between components of different criticality levels. |
| SIL/HIL | Software-in-the-Loop / Hardware-in-the-Loop — simulation environments for safety-critical integration testing. |
| Matrix A (Validation) | Traceability matrix: REQ → ATP → SCN. Proves the system does what the user needs. |
| Matrix B (Verification) | Traceability matrix: REQ → SYS → STP → STS. Proves the architecture works as designed. |
| Matrix C (Integration Verification) | Traceability matrix: SYS → ARCH → ITP → ITS. Proves the module boundaries work correctly. SYS cells include parent REQ references. |
