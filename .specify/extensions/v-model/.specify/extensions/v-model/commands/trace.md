## Goal

Build a **regulatory-grade Bidirectional Requirements Traceability Matrix (RTM)** that provides a complete, auditable trail between every requirement, its test cases, and its executable scenarios. This is the single most scrutinized document during a compliance audit.

Industry standards — **DO-178C** (Aviation), **ISO 26262** (Automotive), **IEC 62304** (Medical Devices), **FDA 21 CFR Part 820** (Medical Quality Systems), and **IEC 61508** (Industrial Safety) — explicitly dictate what a compliant traceability artifact must contain. Our matrix must meet these standards.

This is a **read-only analysis** command — it generates a report but does NOT modify `requirements.md` or `acceptance-plan.md`.

## The 4 Pillars of Regulatory-Grade Traceability

Every RTM generated by this command MUST satisfy these 4 pillars:

### Pillar 1: Strict Bidirectionality (The Golden Rule)

Auditors require **bidirectional traceability** — not just forward, but also backward:

- **Forward Traceability (REQ → ATP → SCN):** Proves every requirement has been tested. *Ensures no missing features.*
- **Backward Traceability (SCN → ATP → REQ):** Proves every test maps back to a justified requirement. *Ensures no gold-plating, undocumented features, or rogue code (a massive security/safety risk).*

Both directions are checked and reported.

### Pillar 2: Orphan & Gap Analysis

The RTM doesn't just show what IS connected — it explicitly highlights what is BROKEN:

- **Gaps:** A `REQ` with no `ATP`. The system is incomplete — untested functionality.
- **Orphans:** An `ATP` or `SCN` with no `REQ`. The system has undocumented behavior — a compliance failure.

The script actively scans for and flags both gaps and orphans in a dedicated section.

### Pillar 3: Versioning and Baselines

When requirements change, linked tests become "suspect" until re-verified. The matrix includes:

- **Artifact hashes or timestamps** proving the matrix reflects the CURRENT requirements and test plans, not a stale draft.
- **Source file references** with modification dates for audit trails.

(Deep git-diff-based version tracking is a Phase 2 feature; for the MVP, timestamps and file modification dates establish the baseline.)

### Pillar 4: Granular Execution State

A matrix that only says "this test exists" is useless. Each entry must show its verification status:

- `⬜ Pending Execution` — Test case exists but has not been run
- `✅ Passed` — Test executed and passed (future: linked to CI/CD results)
- `❌ Failed` — Test executed and failed
- `🚫 Blocked` — Test cannot be executed due to a dependency
- `⏸️ Deferred` — Test intentionally deferred with justification

For the MVP, all items default to `⬜ Pending Execution`. Future phases will pull status from CI/CD pipelines or test runners.

## Execution Steps

### 1. Setup

Run `{SCRIPT}` from the repository root. The `--require-reqs --require-acceptance` flags ensure both prerequisite files exist.

Parse the JSON output for:
- `VMODEL_DIR`: Path to `specs/{feature}/v-model/` directory
- `REQUIREMENTS`: Path to `requirements.md`
- `ACCEPTANCE`: Path to `acceptance-plan.md`

**Note on script paths**: The helper scripts (`build-matrix`) are in the same directory as the setup script. Derive the scripts directory from the `{SCRIPT}` path (i.e., its parent directory).

For single quotes in args like "I'm Groot", use escape syntax: e.g 'I'\''m Groot' (or double-quote if possible: "I'm Groot").

### 2. Build the Matrix (Deterministic Script)

Run the matrix builder script — this is a **deterministic operation**, NOT an AI task:

```bash
{SCRIPTS_DIR}/build-matrix.sh {VMODEL_DIR} --output {VMODEL_DIR}/traceability-matrix.md
```
```powershell
{SCRIPTS_DIR}/build-matrix.ps1 {VMODEL_DIR} -Output {VMODEL_DIR}/traceability-matrix.md
```

Where `{SCRIPTS_DIR}` is the parent directory of the `{SCRIPT}` path.

The script:
1. Parses `requirements.md` for all REQ IDs and their descriptions
2. Parses `acceptance-plan.md` for all ATP IDs, descriptions, and linked SCN IDs
3. Cross-references forward (REQ → ATP → SCN) and backward (SCN → ATP → REQ)
4. Identifies gaps (uncovered REQs) and orphans (unlinked ATPs/SCNs)
5. **If `system-design.md` and `system-test.md` exist**, generates Matrix B (Verification) alongside Matrix A (Validation) — parsing SYS components from the Decomposition View and STP/STS test cases
6. Generates the complete traceability matrix with coverage metrics
7. Writes output to `{VMODEL_DIR}/traceability-matrix.md`

**Dual-matrix output (v0.2.0+):**
- **Matrix A — Validation (User View):** REQ → ATP → SCN. Always present.
- **Matrix B — Verification (Architectural View):** REQ → SYS → STP → STS. Only present when system-level artifacts exist.
- Backward compatible: v0.1.0 feature directories (no system artifacts) produce Matrix A only.

### 3. Present Results (3 Sections)

Read the generated `traceability-matrix.md` and present the results to the user in the mandatory 3-section format. Your role is to **summarize and interpret** the deterministic output, not to recalculate it.

### 5. Recommend Next Actions

Based on the analysis:

- **If fully compliant (0 gaps, 0 orphans)**:
  "✅ All requirements are bidirectionally traceable to test cases and scenarios. The RTM is ready for audit review. Proceed with `/speckit.plan` to start implementation planning."

- **If gaps exist (untested requirements)**:
  "❌ Forward traceability failures found. Run `/speckit.v-model.acceptance` to generate missing test cases and scenarios for the {N} uncovered requirements listed in the Exception Report. Then re-run `/speckit.v-model.trace` to rebuild the matrix."

- **If orphans exist (unlinked tests)**:
  "⚠️ Backward traceability failures found. {N} test cases or scenarios do not trace to any requirement. Review the orphans listed in the Exception Report — they may indicate deleted requirements. Remove or reassign orphaned items in `acceptance-plan.md`, then re-run `/speckit.v-model.trace`."

- **If both gaps and orphans exist**:
  "❌ Bidirectional traceability failures found in both directions. Address gaps first (run `/speckit.v-model.acceptance`), then clean up orphans manually, then re-run `/speckit.v-model.trace`."

## Operating Constraints

### Read-Only Command

This command MUST NOT modify `requirements.md` or `acceptance-plan.md`. It only writes to `traceability-matrix.md`. If issues are found, it recommends corrective actions but does not execute them.

### Deterministic Output

The matrix is built by a script using regex pattern matching, NOT by the AI. The AI's role is limited to:
1. Running the script
2. Reading and presenting the script's output
3. Interpreting gaps and recommending next actions

This ensures the matrix is 100% accurate and audit-ready — no hallucinated coverage claims.
