# Compliance Guide

This guide explains how the V-Model Extension Pack supports regulatory compliance for safety-critical software development.

## Supported Standards

The V-Model Extension Pack generates artifacts that satisfy traceability requirements from:

| Standard | Domain | Key Traceability Requirement |
|----------|--------|------------------------------|
| **IEC 62304** | Medical Devices | Clause 5.7 — Software verification including traceability of requirements to verification activities |
| **ISO 26262** | Automotive | Part 6, Clause 9 — Verification of software safety requirements through traceability to test cases |
| **DO-178C** | Aviation | Section 6.3.4 — Traceability between requirements, design, code, and test cases |
| **FDA 21 CFR Part 820** | Medical Quality Systems | §820.30(i) — Design validation ensuring devices conform to defined user needs |
| **IEC 61508** | Industrial Safety | Part 3, Clause 7.9 — Software aspects of system safety validation |

## What Auditors Expect

### 1. Requirements Traceability

Auditors verify that every requirement has:
- A **unique, persistent identifier** (our `REQ-NNN` schema)
- A **testable description** (enforced by IEEE 29148 / INCOSE 8-criteria validation)
- A **clear verification method** (Test, Inspection, Analysis, Demonstration)
- **Forward traceability** to at least one test case

**How we satisfy this**: The `/speckit.v-model.requirements` command enforces all 8 quality criteria (Unambiguous, Testable, Atomic, Complete, Consistent, Traceable, Feasible, Necessary) and assigns persistent IDs.

### 2. Test Case Traceability

Auditors verify that every test case:
- **Maps to exactly one requirement** (our `ATP-NNN-X` links to `REQ-NNN`)
- Has a **clear pass/fail condition** (enforced by test case quality criteria)
- Has **executable steps** (our BDD scenarios: `SCN-NNN-X#`)

**How we satisfy this**: The `/speckit.v-model.acceptance` command enforces 4 test case criteria (Traceable, Independent, Repeatable, Clear Expected Result) and 4 scenario criteria (Declarative, Single Action, Strict Preconditions, Observable Outcomes).

### 3. Bidirectional Traceability Matrix

Auditors require proof that:
- **Forward**: Every requirement → has test cases → has executable scenarios (no gaps)
- **Backward**: Every test case → traces to a requirement (no orphans)
- The matrix is **current** (reflects the latest requirement baseline)
- The matrix is **accurate** (not self-assessed — generated by a deterministic tool)

**How we satisfy this**: The `/speckit.v-model.trace` command uses a deterministic script (regex-based, not AI-generated) to build the matrix with:
- Coverage Audit section (the math)
- Exception Report (gaps and orphans)
- Full 3D matrix table
- Baseline Information (timestamps, file hashes, git commit)

### 4. System Design Traceability (IEEE 1016:2009 + ISO 29119-4)

Auditors in safety-critical domains verify that system design decisions are:
- Documented via **IEEE 1016:2009** design views (Decomposition, Dependency, Interface, Data Design)
- Tested via **ISO 29119-4** techniques (Interface Contract Testing, Boundary Value Analysis, Fault Injection, Equivalence Partitioning)
- Traceable using the `SYS-NNN` → `STP-NNN-X` → `STS-NNN-X#` ID schema

**How we satisfy this**: The `/speckit.v-model.system-design` command generates design elements aligned with IEEE 1016 views. The `/speckit.v-model.system-test` command generates test procedures mapped to ISO 29119-4 techniques.

### 5. Architecture Design Traceability (IEEE 42010 + ISO 29119-4)

Auditors in safety-critical domains verify that architecture decisions are:
- Documented via **IEEE 42010** / **Kruchten 4+1** architecture views (Logical, Process, Interface, Data Flow)
- Tested via **ISO 29119-4** integration techniques (Interface Contract Testing, Data Flow Testing, Interface Fault Injection, Concurrency & Race Condition Testing)
- Traceable using the `ARCH-NNN` → `ITP-NNN-X` → `ITS-NNN-X#` ID schema
- Decomposed per **ISO 26262-9 §5** when ASIL allocation applies

**How we satisfy this**: The `/speckit.v-model.architecture-design` command generates architecture elements aligned with IEEE 42010 views. The `/speckit.v-model.integration-test` command generates integration test procedures mapped to ISO 29119-4 techniques. CROSS-CUTTING modules are validated across all dependent modules.

### 6. Module Design Traceability (DO-178C Low-Level Requirements + ISO 26262 Part 6)

Auditors in safety-critical domains verify that module-level designs are:
- Documented with **four mandatory views** (Algorithmic/Logic, State Machine, Internal Data Structures, Error Handling & Return Codes)
- Decomposed from architecture elements (`MOD-NNN` decomposes `ARCH-NNN`)
- Tested via **white-box techniques** (Statement & Branch Coverage, Boundary Value Analysis, Strict Isolation, State Transition Testing)
- Traceable using the `MOD-NNN` → `UTP-NNN-X` → `UTS-NNN-X#` ID schema
- Unit tests enforce **strict isolation** with explicit Dependency & Mock Registries per test procedure

**How we satisfy this**: The `/speckit.v-model.module-design` command generates module designs with pseudocode, state machines, data structures, and error handling. The `/speckit.v-model.unit-test` command generates white-box unit test procedures with strict isolation — every external dependency is mocked via Dependency & Mock Registries. Modules tagged `[EXTERNAL]` or `[CROSS-CUTTING]` are handled with appropriate bypass rules.

### 7. Multi-Matrix Traceability

Auditors require end-to-end traceability across *all* V-Model levels, not just one. The multi-matrix approach provides:
- **Matrix A** (Requirements → Acceptance): `REQ-NNN` → `ATP-NNN-X` → `SCN-NNN-X#` — validates *what* was requested
- **Matrix B** (System Design → System Testing): `SYS-NNN` → `STP-NNN-X` → `STS-NNN-X#` — validates *how* it was designed
- **Matrix C** (Architecture → Integration Testing): `SYS-NNN` → `ARCH-NNN` → `ITP-NNN-X` → `ITS-NNN-X#` — validates *how modules interact*
- **Matrix D** (Module Design → Unit Testing): `ARCH-NNN` → `MOD-NNN` → `UTP-NNN-X` → `UTS-NNN-X#` — validates *how each module works internally*
- **Matrix H** (Hazard → Mitigation Verification): `HAZ-NNN` → mitigation (`REQ-NNN`/`SYS-NNN`) → verification (`ATP`/`STP`) — validates *risk is mitigated and verified*

**How we satisfy this**: The `/speckit.v-model.trace` command generates all applicable matrices in a single report with independent coverage audits. Matrix H is automatically included when `hazard-analysis.md` exists. An auditor sees one consolidated artifact proving traceability at all levels.

### 8. Hazard Analysis and Risk Management

Regulated industries (especially medical devices and automotive) require proof that hazards have been identified, assessed, and mitigated. ISO 14971 (Medical Risk Management) and ISO 26262 Part 9 (Automotive HARA) mandate a Failure Mode and Effects Analysis (FMEA).

**How we satisfy this**: The `/speckit.v-model.hazard-analysis` command generates an FMEA register with:
- **Hazard identification** (`HAZ-NNN`) linked to system components (`SYS-NNN`)
- **Operational state awareness** — the same failure is assessed across different system states (e.g., IDLE vs. CRUISE), as required by ISO 14971 and ISO 26262
- **Severity × Likelihood risk matrix** with standardized scales
- **Mitigation traceability** — each mitigation references `REQ-NNN` or `SYS-NNN`, creating a chain: Hazard → Mitigation → Requirement → Test Case
- **Three-dimensional coverage validation** via `validate-hazard-coverage.sh`: forward (every SYS covered), backward (every HAZ mitigation valid), and state consistency

## ID Schema for Audit Trails

The three-tier ID schema (`REQ-NNN` → `ATP-NNN-X` → `SCN-NNN-X#`) is designed for auditability:

1. **Self-documenting lineage**: Reading `SCN-001-A1` immediately tells the auditor it validates `ATP-001-A` which tests `REQ-001`. No lookup table needed.

2. **Permanent IDs**: IDs are never renumbered. If REQ-003 is removed, REQ-004 stays REQ-004. Gaps are acceptable — renumbering would break audit trails.

3. **Category prefixes**: `REQ-NF-001` (Non-Functional), `REQ-IF-001` (Interface), `REQ-CN-001` (Constraint) allow auditors to filter by requirement type.

4. **System-level lineage**: The same scheme extends to the System Design ↔ System Testing level: `SYS-NNN` → `STP-NNN-X` → `STS-NNN-X#`. Reading `STS-002-B3` immediately tells the auditor it validates `STP-002-B` which tests `SYS-002`.

5. **Architecture-level lineage**: The scheme extends further to the Architecture Design ↔ Integration Testing level: `ARCH-NNN` → `ITP-NNN-X` → `ITS-NNN-X#`. Reading `ITS-003-A2` immediately tells the auditor it validates `ITP-003-A` which tests `ARCH-003`.

6. **Module-level lineage**: At the bottom of the V, the scheme completes with Module Design ↔ Unit Testing: `MOD-NNN` → `UTP-NNN-X` → `UTS-NNN-X#`. Reading `UTS-001-A1` immediately tells the auditor it validates `UTP-001-A` which tests `MOD-001`.

## Deterministic Validation

A critical compliance requirement: **the traceability matrix must be generated by a deterministic tool, not human judgment or AI self-assessment**.

Our helper scripts (`validate-requirement-coverage.sh`, `validate-system-coverage.sh`, `validate-architecture-coverage.sh`, `validate-module-coverage.sh`, `validate-hazard-coverage.sh`, `impact-analysis.sh`, `build-matrix.sh`) use regex pattern matching to:
- Extract all IDs from markdown files
- Cross-reference coverage in both directions
- Calculate exact metrics
- Flag specific gaps and orphans
- Trace change impact across the full V-Model graph

This means:
- The coverage percentage is **mathematically correct**, not estimated
- An auditor can re-run the same script and get the **same result**
- No risk of AI "hallucinating" 100% coverage when gaps exist

## Change Impact Analysis

When a requirement, design component, or test artifact changes, regulatory standards require that all dependent artifacts be reviewed and re-validated. The `/speckit.v-model.impact-analysis` command automates this traceability traversal.

### How It Works

The script builds a dependency graph from all V-Model markdown artifacts in the feature directory, then traverses it in one of three modes:

| Mode | Flag | Use Case |
|------|------|----------|
| Downward | `--downward` | A requirement changed — what tests and modules are now suspect? |
| Upward | `--upward` | A module or test failed — which requirements are at risk? |
| Full | `--full` | Bidirectional — show the complete blast radius in both directions |

### Example: Requirement Change

```bash
/speckit.v-model.impact-analysis --downward REQ-003 specs/feature-001/v-model/
```

This identifies all suspect artifacts downstream of `REQ-003`: system design components (`SYS-NNN`), architecture modules (`ARCH-NNN`), detailed modules (`MOD-NNN`), hazards (`HAZ-NNN`), and every test at each level. The output includes:

- **Suspect artifacts** grouped by V-Model level
- **Blast radius** (total count and breakdown by level)
- **Re-validation order** (bottom-up for downstream, top-down for upstream)

### Example: Bottom-Up Investigation

```bash
/speckit.v-model.impact-analysis --upward MOD-004 --json specs/feature-001/v-model/
```

When a hardware component goes end-of-life or an integration test consistently fails, the `--upward` flag traces back to the originating requirements so the product owner knows exactly which user-facing capabilities are affected.

### CI Integration

Use `--json` output to integrate impact analysis into CI pipelines:

```bash
# Block PR if blast radius exceeds threshold
blast=$(impact-analysis.sh --json --downward REQ-003 ./v-model/ | python3 -c "import json,sys; print(json.load(sys.stdin)['blast_radius']['total'])")
if [ "$blast" -gt 50 ]; then echo "Blast radius too large for single PR"; exit 1; fi
```

### Audit Value

- **Deterministic**: Same input always produces the same output — auditors can verify independently
- **Complete**: Traverses the entire dependency graph, not just direct references
- **Documented**: Output serves as a change-impact record for the audit trail

## Preparing for an Audit

### Pre-Audit Checklist

1. Run `/speckit.v-model.trace` to generate the current traceability matrix
2. Run `/speckit.v-model.impact-analysis --full <changed-ID>` for any recently modified artifacts
3. Verify "OVERALL STATUS: ✅ COMPLIANT" in the Coverage Audit
4. Verify 0 items in the Exception Report
5. Check the Baseline Information section for current timestamps
6. Ensure all source files (`requirements.md`, `acceptance-plan.md`, `hazard-analysis.md`, etc.) are committed to version control
7. Archive the `traceability-matrix.md` and any `impact-report.md` as audit artifacts

### Common Audit Findings and Prevention

| Finding | Prevention |
|---------|------------|
| Requirement without test case | Run `/speckit.v-model.trace` — gaps are flagged in Exception Report |
| Test case without requirement (orphan) | Run `/speckit.v-model.trace` — orphans are flagged in Exception Report |
| Vague requirement ("the system shall be fast") | Enforced by Criterion 1 (Unambiguous) and Banned Words list in requirements command |
| Compound requirement hiding untested behavior | Enforced by Criterion 3 (Atomic) in requirements command |
| Test case dependent on other test state | Enforced by Criterion 2 (Independent) in acceptance command |
| Non-repeatable test | Enforced by Criterion 3 (Repeatable) in acceptance command |
